✦ AI-Native Marketing CRM

Zari CRM

A production-grade, multi-agent marketing CRM for Indian retail — built on LangGraph, Gemini 2.0 Flash, RAG, and React. Natural language in, executed campaigns out.

LangGraph LangChain Gemini 2.0 Flash React 18 MongoDB Qdrant FastAPI Node/Express Socket.io Docker

01 —Project Overview

Zari CRM is an AI-native marketing platform where AI agents autonomously handle the full campaign lifecycle — from understanding marketer intent in natural language, to segmenting customers, writing personalised messages, selecting optimal channels, dispatching communications, and analysing results. It is a functional replica of Xeno.ai — a production CRM used by Indian and Middle-East retail brands.

What makes it AI-native

🧠

No manual segmentation

Natural language → MongoDB aggregation pipeline generated by the Segmentation Agent using RAG over past segment definitions.

✍️

No template library

Copy generated fresh per campaign, grounded in RAG over brand voice corpus and high-performing past campaigns.

👤

Per-customer personalisation

Every message personalised with real product recommendations, persona-driven offers, and best send time.

🔄

Self-improving loop

Analytics Agent embeds campaign outcomes into Qdrant. Future campaigns automatically learn from past performance.

⏸️

Human-in-the-loop

LangGraph interrupt_before gate pauses execution for campaigns >1000 recipients. Frontend shows approval modal.

📡

Real-time agent streaming

Socket.io pushes each agent's progress to the UI live. Marketer watches the AI work step by step.

💬

Real WhatsApp delivery

Baileys QR-scan login. Campaigns with channel=whatsapp send to real phones. Session persists across restarts. Falls back to simulation if disconnected.

🚨

Autonomous Campaign Monitor

LangGraph monitor graph evaluates running campaigns against channel thresholds. Auto-generates diagnosis + improved copy. Resends to non-openers on approval.

04 —8 AI Agents

Every agent is a LangGraph node. Each uses ChatGoogleGenerativeAI(model="gemini-2.0-flash") with structured output or tool-calling depending on the task.

🧠
Supervisor Agent
Entry point · graph.py → supervisor_node

Parses the marketer's natural language query into a structured CampaignPlan Pydantic model. Decides which sub-agents to invoke and in what order. Uses llm.with_structured_output(CampaignPlan).

Output: intent, steps[], segment_criteria, channel_preference, goal, campaign_name, journey_trigger

Structured outputLangGraph routingNo tools
🎯
Segmentation Agent
segmentation.py · Structured output + direct pipeline execution

LLM generates a typed SegmentPipeline Pydantic model (stages[], segment_name, segment_description) via with_structured_output(). Pipeline executed directly against MongoDB — no unreliable AgentExecutor JSON parsing. Uses RAG over past segment definitions as few-shot examples. Graph exits early if 0 customers matched.

Structured outputRAG: segment_profilesDirect MongoDB executionEarly-exit gate
✍️
Campaign Creation Agent
campaign_creation.py · Chain with structured output

Generates campaign copy with 2 A/B variants. Retrieves top-3 past campaigns from Qdrant (campaign_performance) and brand voice guidelines (brand_voice_corpus). Output: CampaignCopy with headline, body, cta, variants[].

Structured outputRAG: campaign_performanceRAG: brand_voice_corpus
🪪
Persona Agent
persona_agent.py · RFM + LLM labeling

Computes RFM (Recency, Frequency, Monetary) scores for all customers and classifies them into 7 segments: Champions, Loyal, At Risk, Cannot Lose, Lost, New, Potential. Uses LLM to generate evocative persona labels like "Fashion-Forward Loyalist".

RFM scoringLLM labelingMongoDB upsert
👤
Personalisation Agent
personalization.py · asyncio.gather() batch

Personalises every message using customer profile, real product recommendations from the product catalog, persona-driven offers selected by select_best_offer tool (respects budget caps and offer sensitivity), and best send time from CustomerPersona.

Async batchoffer_toolspersona_toolsMongoDB products
📡
Channel Selection Agent
channel_selection.py · Persona-driven

Assigns optimal channel per customer. Priority: (1) CustomerPersona.engagement.best_channel with confidence score, (2) historical open rates from communications, (3) customer opt-in preferences, (4) fallback to email. Enriches assignments with best_send_hour.

Persona-drivenEngagement historyChannel multipliers
🚀
Execution Agent
execution.py · Batched dispatch

Dispatches personalised messages to the Channel Service in batches of 50. Saves Communication docs to MongoDB. Handles retries on failure. The Channel Service simulates probabilistic delivery lifecycle and fires webhooks back.

Batched HTTPCommunication recordsRetry logic
📊
Analytics + Optimisation Agents
analytics.py + optimization.py

Analytics: aggregates funnel metrics (sent→delivered→opened→clicked→converted), generates natural language insight report via LLM, and stores outcome narrative back into Qdrant (the self-learning loop). Optimisation: selects A/B winner, recommends optimal send window, suggests segment refinements.

MongoDB aggregationRAG store-backA/B winner

⏸️

Human approval gate — Graph compiled with interrupt_before=["await_approval"]. When campaign exceeds 1000 recipients, LangGraph pauses, emits agent:progress with requires_approval: true via Socket.io. Frontend shows approval modal. Marketer approves or rejects. Backend calls POST /run/:id/resume → compiled_graph.update_state() → graph resumes.

06 —RAG Implementation

Retrieval-Augmented Generation grounds LLM outputs in real data — preventing hallucinated brand voice, re-using proven campaign angles, and improving MongoDB query generation.

Embedding Model

🔢

sentence-transformers/all-mpnet-base-v2 — 768-dimensional vectors, runs locally on CPU, free, strong semantic quality for retail/marketing text. Loaded via langchain-huggingface.HuggingFaceEmbeddings.

Campaign Creation RAG Flow

1
Query construction

Agent builds query: "re-engagement loyalty WhatsApp lapsed customers fashion India"

2
Qdrant similarity search

Top-3 past campaigns retrieved from campaign_performance by cosine similarity (768d vectors).

3
Brand voice retrieval

Separate query to brand_voice_corpus fetches Zari's tone guidelines and approved copy examples.

4
Grounded generation

Retrieved docs injected as context into the campaign creation prompt. LLM generates copy grounded in proven angles.

5
Self-learning store-back

After campaign runs, Analytics Agent embeds outcome narrative back into campaign_performance. Next similar campaign automatically learns from this result.

11 —Offers Engine

AI selects the best offer per customer using the select_best_offer tool, which factors in:

  • offer_sensitivity — from CustomerPersona (0–1). High sensitivity → show higher % off
  • rfm_segment — Champions get VIP offers; churned customers get winback offers
  • budget caps — offer skipped if budget.spent ≥ budget.total_budget
  • uses_per_customer — enforced at code generation and validation

14 —WhatsApp Integration

Real WhatsApp delivery via @whiskeysockets/baileys — no Puppeteer, no Meta Business API. QR-scan login, session persisted to disk, survives service restarts.

Architecture

Frontend (/whatsapp) ──Socket.io──► whatsapp-service :3003 │ Baileys WA client │ auth_info_baileys/ (session) Backend :3001 ──proxy──► GET /status · GET /qr · POST /disconnect channel-service :3002 ──► POST /send ← real phone number E.164 ↑ fallback to simulation if :3003 unreachable

QR Login Flow

  1. 1
    Start whatsapp-service → Baileys emits whatsapp:qr via Socket.io
  2. 2
    Frontend /whatsapp page displays QR image (base64)
  3. 3
    Scan with phone → WhatsApp → Settings → Linked Devices → Link Device
  4. 4
    Baileys emits whatsapp:ready with connected phone number → status badge turns green
  5. 5
    Session saved to ./auth_info_baileys/ — survives restarts, no re-scan needed

Phone Number Normalization

Customer phones stored as 10-digit Indian strings ("9876543210"). Baileys needs E.164 JID format.

// whatsapp-service/src/utils/phone.js
function toWhatsAppJid(phone) {
  const digits = phone.replace(/\D/g, "");
  const e164 = digits.length === 10 ? `91${digits}` : digits;
  return `${e164}@s.whatsapp.net`;
}

Channel Service Routing

When channel === "whatsapp" and the execution node has a real phone, it sends to POST /send on the WhatsApp service. On ECONNREFUSED or non-200, falls back to channel-service simulation — so the rest of the pipeline is unaffected when WhatsApp service is offline.

📱

Real sends require a linked WhatsApp account. Sessions expire if the linked device is removed from WhatsApp settings. The service auto-reconnects on restart if auth_info_baileys/ exists.

Alert Actions
